📝 Notas Técnicas
Padrões de Design do Modelo
PadrãoImplementação
Filtro por período (De/Até) d_Período1 e d_Período2 com relacionamento bidirecional Many:Many via ano.mes na d_Calendario. Permite seleção de intervalo nos relatórios.
Budget com TREATAS Medidas de budget usam TREATAS + d_Mascara_DRE para propagar filtros de linha DRE corretamente — evita uso direto de f_Budget[Premissas].
ISFILTERED vs COUNTROWS Medidas de período usam ISFILTERED(...) para distinguir seleção explícita de slicer de filtros de página — mais confiável que COUNTROWS(VALUES(...)).
CSS em DAX — SUBSTITUTE Medidas HTML usam SUBSTITUTE(FORMAT(valor,"#,##0.0"), ",", ".") para evitar que o separador decimal DAX quebre atributos CSS como width.
HTML Cards — bloco VALORES Medidas HTML seguem padrão: declaração de variáveis no bloco // ====== VALORES ====== no topo para reusabilidade e legibilidade.
Detecção automática de mês Medidas v2 detectam automaticamente o último mês com dado realizado — eliminam necessidade de seleção manual no slicer de calendário.
DirectQuery — impacto Todo cálculo é executado em tempo real contra o IFS Oracle. Evitar medidas com COUNTROWS(ALL(...)) em larga escala. Preferir ISFILTERED e SELECTEDVALUE.
⚠ Atenção ao renomear colunas: Por ser base DirectQuery de 8 relatórios downstream, qualquer renomeação de coluna ou remoção de tabela quebra os relatórios conectados. Sempre use aliás em Power Query e valide impacto antes de publicar.
